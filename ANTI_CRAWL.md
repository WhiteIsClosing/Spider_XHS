# 反爬对抗使用指南（单 IP / 单 Cookie 场景）

本指南面向**没有代理池、只能手动提供一个 Cookie**的使用场景。
此时无法靠"换 IP / 换账号"分摊风险，唯一可行的策略是 **把这一个身份保护到极致**：

1. **不被发现** —— 指纹内部一致、节奏像真人、单日量受控；
2. **被发现立刻撤** —— 命中风控就退避，连续命中就熔断停跑，绝不硬撞；
3. **撑得久** —— 滚动并持久化 Cookie，让这唯一的身份"自然变老"。

下面先讲 **你需要做什么**，再列出 **全部可调开关** 和 **运行中如何看日志**。

---

## 一、你需要做什么（操作清单）

### 1. 安装依赖（`curl_cffi` 必须装好）

```bash
pip install -r requirements.txt
npm install        # 签名算法需要 Node.js 20+
```

> ⚠️ **`curl_cffi` 一定要装成功**。它负责把 TLS / HTTP2 指纹伪装成真实 Chrome，是对抗
> WAF 指纹的最关键一层。如果它没装上，程序会退回普通 `requests`，TLS 指纹会"长得像 Python"，
> 几乎必被识别。启动日志里确认这一行：
>
> ```
> [HTTP] backend=curl_cffi impersonate=chrome120
> ```
>
> 如果看到 `backend=requests-fallback`，说明 `curl_cffi` 没装好，请先解决。

### 2. 提供 Cookie（手动）

在项目根目录 `.env` 写入登录后的 Cookie：

```
COOKIES='你的cookie字符串'
```

Cookie 必须包含 `a1`、`web_session`、`webId` 三个字段（程序会校验）。
获取方式见 README 的"配置 Cookie"一节。

> 这个 Cookie 是你唯一的身份资产，请像保护密码一样保护它。

### 3. 运行

```bash
python -m spider.spider
```

启动后程序会依次做：**签名工具链自检 → 配置预算/单日配额 → 校验 Cookie/签名 → 预热（逛首页）→ 正式采集**。

### 4. 看懂日志里的关键信号

| 日志前缀 | 含义 | 你该怎么做 |
|---|---|---|
| `[HTTP] backend=curl_cffi` | 指纹后端正常 | 无需操作（若是 fallback 要修） |
| `[Pipeline] 签名工具链自检通过` | 签名 JS 能跑 | 无需操作 |
| `[限速] ... 冷却 / 长尾停顿` | 正常的拟人节流 | 无需操作，**别去缩短延迟** |
| `[风控] 命中风控信号 第 N/3 次，退避 Xs` | 服务端开始限你，正在退避 | 观察；若频繁出现说明该歇了 |
| `[风控] 连续 3/3 次 → 熔断，停止本次运行` | 连续命中风控，已自动停跑 | **停手，等几小时再跑**，别立刻重试 |
| `[Budget] daily quota ... today used=N` | 今日已用请求数 | 临近上限说明今天该收工 |
| `Cookie 已失效` | 登录态过期 | 重新获取 Cookie 填回 `.env` |
| `签名疑似过期，请更新 static/ 下的签名 JS` | 小红书改了签名算法 | 更新 `static/` 下签名脚本 |

### 5. 出问题时怎么办

- **触发熔断（circuit breaker）**：这是保护机制，不是 bug。说明小红书已经在限制你。
  **不要立刻重跑**——立刻重试只会把"软封"升级成"硬封"，烧掉你的 Cookie。建议**间隔数小时**、
  在白天时段、用更长的延迟再试。
- **触发单日配额**：今天这个身份发的请求够多了，明天再来。
- **Cookie 失效**：重新登录获取新 Cookie。
- **签名过期**：更新 `static/` 目录下的签名 JS（小红书会不定期改算法）。

## 二、可调开关（环境变量）

全部写在 `.env` 或运行环境里即可。**默认值已偏保守，建议先按默认跑，再据日志微调。**

### 指纹 / 传输层

| 变量 | 默认 | 说明 |
|---|---|---|
| `XHS_HTTP_BACKEND` | `curl_cffi` | HTTP 后端；保持 `curl_cffi` |
| `XHS_IMPERSONATE` | `chrome120` | 伪装的浏览器指纹；UA/sec-ch-ua 会**自动据此对齐** |
| `XHS_USER_AGENT` | （自动） | 覆盖 UA；**改了它请同时把 `XHS_IMPERSONATE` 对齐到同浏览器同版本**，否则又会指纹打架 |

### 风控熔断（被发现就撤）

| 变量 | 默认 | 说明 |
|---|---|---|
| `XHS_RISK_CIRCUIT_THRESHOLD` | `3` | 连续命中多少次风控信号就熔断停跑 |
| `XHS_RISK_BACKOFF_BASE` | `30` | 首次命中的退避秒数（之后指数翻倍） |
| `XHS_RISK_BACKOFF_CAP` | `300` | 单次退避上限秒数 |
| `XHS_RISK_CODES` | （空） | 额外的业务风控码，逗号分隔，如 `461,10001` |

### 拟人节奏 / 昼夜节律

| 变量 | 默认 | 说明 |
|---|---|---|
| `XHS_CIRCADIAN` | `1` | 是否启用昼夜节律（`0` 关闭） |
| `XHS_ACTIVE_HOUR_START` | `9` | 活跃时段起点（本地 24h 制） |
| `XHS_ACTIVE_HOUR_END` | `23` | 活跃时段终点 |
| `XHS_NIGHT_FACTOR` | `4.0` | 深夜（1–7 点）延迟放大倍数 |
| `XHS_SHOULDER_FACTOR` | `2.0` | 早晚过渡时段延迟放大倍数 |

### 量级控制（代码参数，在 `spider.spider` 调用里传）

| 参数 | 默认 | 说明 |
|---|---|---|
| `max_requests` | `80` | 单次运行最大请求数 |
| `max_minutes` | `30` | 单次运行最大墙钟分钟 |
| `max_requests_per_day` | `300` | **单个身份单日最大请求数（跨重启累计，重启不能绕过）** |
| `delay_range` | `(5,15)` | 每次请求间隔（秒），走 lognormal 长尾 |
| `cooldown_every` / `cooldown_range` | `10` / `(60,120)` | 每 N 次请求触发一次长冷却 |

## 三、运行时会自动生成的文件

| 文件 | 作用 | 是否提交 |
|---|---|---|
| `.xhs_cookies_live` | 滚动后的 Cookie（含 `web_session` 等敏感值） | ❌ 已在 `.gitignore`，**切勿提交** |
| `.xhs_quota.json` | 单日请求计数（跨重启） | ❌ 已在 `.gitignore` |

> 下次运行时，如果 `.env` 里的 Cookie 与上次是**同一身份**（`a1` 相同），程序会自动复用
> `.xhs_cookies_live` 里"养老过"的版本，让身份更连贯。如果你换了新账号的 Cookie，会自动用新的。

## 四、红线（别做这些，否则前功尽弃）

- ❌ **别缩短延迟 / 关掉昼夜节律去抢速度**——单 IP 下速度就是风险。
- ❌ **熔断后别立刻重跑**——给服务端的风险评分降温的时间，否则升级硬封。
- ❌ **别并发 / 多线程跑同一个 Cookie**——当前为单线程拟人设计，并发=自杀。
- ❌ **别把 `.xhs_cookies_live` 提交到仓库**——里面是你的登录态。
- ✅ 想加大采集量时，正确做法是**拉长时间、分多天**，而不是单日猛冲。

## 五、底层做了什么（简述）

- **Tier 1**
  - 风控信号闭环：识别 HTTP 429/461/47x 与业务 `频繁/验证/风控` 等信号 → 指数退避 → 连续命中熔断停跑（控制信号继承 `BaseException`，不会被 API 层 `except Exception` 吞掉）。
  - 指纹一致：UA + Client Hints 由 `XHS_IMPERSONATE` 推导并全程固定，与 TLS 指纹一致，不再每请求随机换 UA。
  - Cookie 滚动 + 持久化：把响应里的 `Set-Cookie`（`acw_tc`/`web_session`）回写后续请求并落盘；仅限 `xiaohongshu.com` 域，媒体流不受影响；`a1` 不变，签名仍有效。
- **Tier 2**
  - 跨运行单日配额：单日请求数落盘，重启不能绕过。
  - 昼夜节律：所有停顿按时段放大，深夜显著放慢。
  - 签名新鲜度自检：启动即离线验证签名 JS 能否跑通；运行前再以一次轻量请求确认签名未被服务端拒绝，疑似过期则提前报错而不是硬撞。

> 仍然缺失的最大短板是 **没有 IP/代理池**——这是单 IP 场景的天花板，本项目通过"保守 + 退避 + 熔断"把单身份用到尽量稳，但无法突破单 IP 的根本限制。
